## 🧐 課題1: AIライターのジャンル誤認（ハルシネーション）バグ

### 【事象・コンテキスト】
`auto_post.py` および `novelove_writer.py` において、データベース側のカテゴリー・タグは `comic_tl`（TL漫画）として正しく識別されているにも関わらず、生成された記事本文の冒頭で「今回はこの **BL漫画** を紹介します」といった致命的なジャンル誤認が発生している。

### 【根本原因の特定】
1. 現在の `novelove_writer.py` -> `build_prompt` において、AIに渡すシステムプロンプト内に「作品の対象ジャンル（BLかTLか）」を明示する情報が存在しない。
2. 担当ライターの人格は10%の確率で「ゲスト枠」として専門外のキャラクター（例：BL専門の「葵」がTL作品を担当する等）が選任される。
3. AIは自身のキャラクター設定（熱狂的なBL好き等）と、あらすじ情報のみから推測を行うため、「自分が担当するのだからこれもBL作品なのだろう」と誤認し、BL文脈の語り口でTL作品を評価してしまう。

### 【提案する解決策】
`novelove_writer.py` の `build_prompt` 内の `【対象作品情報】` セクションに、以下の通りジャンルを強制的に明記する。
```diff
 【対象作品情報】
 タイトル: {safe_title}
 あらすじ: {safe_desc}
+{f"作品ジャンル: {_genre_label(target.get('genre', ''))}"}
```
**[Review AI への指示]**: プロンプト側に「作品ジャンル」を明記するだけで、推論モデル（DeepSeek）の文脈誤認（ハルシネーション）を根本解決できるか、あるいはキャラクター設定とのペルソナ衝突を回避するための更なるプロンプト記述（例: 「今回はあなたの嗜好とは違うジャンルですが～」等の強制補正）が必要か精査してください。

## 🧐 課題2: DigiKet `target=8` における TL 作品の BL 誤分類

### 【事象・コンテキスト】
`novelove_fetcher.py` の `_classify_digiket_genre(genre_tags, target_id)` 関数において、DigiKet の商業BL・TLチャンネル（`target=8`）から取得した作品が、一律で `comic_bl` として処理されている。

### 【根本原因の特定】
2026年3月の v11.4.14 リリース時、`target=8` は「BLが大半を占める」という理由で、以下の通り判定を省略してハードコードされていた。
```python
if target_id == "8":
    return "comic_bl"
```
今回、サーバー上で最新の実データを100件解析した結果、`target=8` 내にTLキーワード（ティーンズラブ、乙女等）を持つ作品が**確実かつ多数混在している**ことが実証されたため、アフィリエイトの機会損失を引き起こしている。

### 【提案する解決策】
他の `target` と同様に、DigiKetから抽出したジャンルタグ文字列 `tags_str` 内に `TL_KEYWORDS` が存在するかを判定し、分岐させる。
```python
if target_id == "8":
    if any(kw in tags_str for kw in TL_KEYWORDS):
        return "comic_tl"
    return "comic_bl"
```
**[Review AI への指示]**: このアプローチの安全性と網羅性を評価してください。対象が「BL・TL混合チャンネル」である以上、デフォルトをBLにフォールバックさせるこのロジックでビジネスロジックとして十分か検証してください。

## 🧐 課題3: スコア3判定ロジックの完全なデッドコード（技術的負債）化

### 【事象・コンテキスト】
`novelove_writer.py` の `build_prompt` 内の `html_structure` 分岐や、生成後タグ補完の `_inject_score3_osusume` 関数など、数十行にわたる「スコア3（低品質あらすじ）」向けの特殊処理が残存している。

### 【根本原因の特定】
v15.2 以降、`auto_post.py` 側で事前審査を導入し、「期待値スコアが規定未満（スコア3相当）の作品は即座に status='excluded' とし、DeepSeek APIの呼び出し（AI執筆）自体を行わない」という仕様に変更された。
これにより、`novelove_writer.py` におけるスコア3対応の生成ロジックには実運用上**絶対に到達しなくなった**。

### 【提案する解決策】
可読性の向上と、将来の仕様変更時における予期せぬ副作用（バグ）の抑止を目的とし、当該コード群を「デッドコード」と見なして完全に削除する。復活が必要になった場合はコミット履歴からサルベージする仕様とする。

**[Review AI への指示]**: AI生成システムにおける「到達不能なフォールバック用ハルシネーション対策コード」を削除することのトレードオフ（将来、事前審査の閾値を下げた場合にシステムがクラッシュするリスクと、現状のメンテナンス負荷の低減）について評価し、本削除提案に対する最終的な「Go」か「No-Go」の判断を下してください。
