# 🐛 次回修正予定のバグと仕様変更（v15.3.1に向けて）

本ドキュメントは、v15.3.0デプロイ後のコードレビューで発覚した「将来のバグの火種」および「アフィリエイトの機会損失」について、原因と修正方針をまとめたものです。
ご自宅でのじっくりとした作業・確認用にご活用ください。

---

## 1. AIのジャンル誤認バグ（ハルシネーション）の修正 🚨重要

### 【症状】
カテゴリーやタグは「TL漫画」と正しくWordPressに登録されているのに、記事本文の冒頭でAIが「今回はこの**BL漫画**を〜」と書いてしまう現象が発生している。

### 【原因】
1. 現在の仕様では、AIに渡すプロンプトに「この作品のジャンル（BLかTLか）」を**明確に伝えるパラメーターが存在していません**。
2. そのため、TL漫画が（10%の確率のゲスト枠などで）**BL専門のキャラクター（紫苑や葵等）に割り当てられた場合**、AIが「自分はBL専門のオタクだ」という自身の設定とあらすじから、「私が紹介するのだからこれもBL漫画のはずだ」と誤認（ハルシネーション）してしまいます。

### 【修正方針】
`novelove_writer.py` の `build_prompt` 関数において、AIへ渡す【対象作品情報】のテキスト内に、明示的にジャンル名（例：`作品ジャンル: TL漫画`）を挿入します。
* 修正箇所: `novelove_writer.py`
* 修正内容: `【対象作品情報】` のセクションに `{_genre_label(target.get("genre", ""))}` を利用した1行を追加し、AIに作品ジャンルを認識させる。

## 2. DigiKetの「商業BL・TLチャンネル（target=8）」全件BL扱いの修正

### 【症状】
DigiKetの「商業BL・TLチャンネル（target=8）」から取得した作品が、TL作品であってもすべて `comic_bl` としてDBに保存され、BL漫画として分類されてしまう。

### 【原因】
* 実装日: 2026年3月27日（v11.4.14 のモジュール分割時）
* 当時の `_classify_digiket_genre` 実装にて、「target=8 はBL作品が大半を占める」という理由から、処理を簡略化して一律で `"comic_bl"` を返すように定義されていました。
* スクリプトによる直近100件の再調査の結果、実際には**ターゲット内にTL作品（ティーンズラブ属性）も多数混在**していることが確認されました。

### 【修正方針】
他のチャンネル（target=6や2）と同様に、`target=8` の作品についてもジャンルタグをパースし、「TLキーワード（ティーンズラブ、乙女など）」が含まれているかを確認するロジックへ変更します。
* 修正箇所: `novelove_fetcher.py`
* 修正内容: `if target_id == "8":` のブロック内で `TL_KEYWORDS` を判定し、該当すれば `comic_tl` を返し、そうでなければ（従来通り大半がBLのため） `comic_bl` を返すよう分岐を追加。

## 3. 到達不能な「スコア3生成ロジック」の完全撤去

### 【症状】
`novelove_writer.py` 内部に、記事品質「スコア3」向けの `html_structure` 分岐や、おすすめタグ不足時に動作する補完機能（`_inject_score3_osusume`）などの複雑なロジックが大量に残っている。

### 【原因】
* v15.2 以降、システムは「スコア3以下の作品は事前審査で破棄し、AIによる執筆API自体を呼ばない」という堅牢な仕様に変更されました。
* これにより、ライター側のスコア3処理は**絶対に到達しないデッドコード（死にコード）**となっています。
* 到達しないコードを巨大なまま放置することは、将来的な仕様変更・機能追加の際に「予期せぬ挙動を引き起こすバグの火種」となります。

### 【修正方針】
使われなくなったスコア3専用の全てのロジックをコード上から完全に削除し、ソースコードをクリーンな状態に保ちます。将来もし足切りラインを下げてスコア3を復活させる場合が来たら、その時に Git の変更履歴（コミットログ）からコードを復元します。
* 修正箇所: `novelove_writer.py`
* 修正内容: スコア3向けのプロンプト構造、及び `_inject_score3_osusume` 関数とその呼び出し部分の全削除。

## 4. 変数名の不統一の解消（マイナー修正）

### 【症状】
`auto_post.py` 内の処理で、`product_id` を参照する際に変数 `pid` と `row['product_id']` が混在している。

### 【原因】
* 今回の例外処理（S-3）の堅牢化の過程で、関数冒頭で `pid = row['product_id']` と宣言したものの、関数中腹の事前キーワードフィルタ部分（`UPDATE`クエリ実行時）で旧来の `row['product_id']` 参照が残ったままになっていました。

### 【修正方針】
* 修正箇所: `auto_post.py`
* 修正内容: `_execute_posting_flow` の L558 付近にて、`(row['product_id'],)` を `(pid,)` に置換し、変数のスコープを統一する。
