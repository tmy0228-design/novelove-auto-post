# Novelove Nexus システム マスター計画書 v5.1

本ドキュメントは、Novelove サイトの記事品質を高い状態に保ち続けるための「統合管理・リライトシステム」の設計図です。
v5.0 では、これまでの議論を統合し、**ユーザー主導の確認→即実行モデル**と**段階的な機能拡張**を基本方針として再構成しました。
v5.1 では、引き継ぎに必要な細部の補完と整合性の修正を実施しました。

---

## 設計の基本原則

1. **検知は自動、実行はユーザー主導**: あらすじ更新や死に記事の検知はシステムが自動で行うが、リライト・パージなどの操作はユーザーが確認画面を経て明示的に実行する。
2. **即実行モデル**: リライトにキューイングは使わない。確認画面→実行ボタン→即反映。キューが必要なのはタグ一括置換など大量処理のみ。
3. **既存機能への無破壊**: `auto_post.py` への変更はオプション引数の追加のみ。引数なしの場合は従来通り動作し、既存の自動投稿に影響しない。
4. **desc_score は不変**: `desc_score` は「取得元サイトのあらすじ充実度」を表す値であり、記事自体の品質スコアではない。リライトしても変更しない。

## フェーズ1: 完全自動のセール検知＆売れ筋タグ管理（実装完了・稼働中）

`nexus_revive.py` により、各プラットフォーム（DLsite, FANZA, DigiKet）を12時間間隔で巡回し、以下を自動管理：
- **割引率30%OFF以上**の作品 → 「🔥期間限定セール」タグ（slug: `sale`）を付与
- ランキング上位の作品 → 「🏆売れ筋作品」タグ（slug: `best-seller`）を付与
- セール終了・ランキング圏外になった作品 → 自動でタグを剥奪
- **これらのタグはリライトエンジンの管轄外**。リライト時に誤って削除しないよう保護する設計。

- **現状**: Read-Only版（閲覧・検索・フィルタリング）の `nexus_dashboard.py` が実装済み

### ダッシュボードの基本設計方針
- **データの「見える化」が最優先**。記事の状態、タグ、スコア、あらすじ変更の有無などを一目で把握できるUIを整備する。
- **リライトなどの操作は、確認画面を経てユーザーが明示的に実行**する。完全自動ではなく、ユーザーの判断を中心に据える。

### リライトエンジン（`nexus_rewrite.py`）

#### 基本フロー（確認→即実行モデル）
1. ダッシュボード上の記事一覧から、ユーザーがリライトしたい記事を選択
2. **確認画面**が表示される：
   - 現在の記事情報（タイトル、公開日、スコア、文字数、担当ライターなど）
   - リライト設定：担当ライター（指定 or ランダム）、感情モード（指定 or ランダム）
   - 「この内容でリライトを実行します」というサマリー表示
3. ユーザーが「実行」を押す → AI再執筆 → WP即反映 → タグ再構築 → DB更新
4. 結果画面：新しい文字数、新タグ、担当ライター名を表示
5. 万が一記事が劣化した場合 → **WordPress標準のリビジョン機能**で旧版に復元（専用のロールバック機構は不要）

#### ライター一覧（`novelove_soul.py` で定義）
リライト時に指定可能なライターID一覧：

| ID | 名前 | 専門ジャンル | 性格 |
|---|---|---|---|
| `shion` | 紫苑 | BL全般 | クールで毒舌な腐女子OL |
| `aoi` | 葵 | BL全般 | 限界オタクの大学生 |
| `ren` | 蓮 | BL全般 | 眼鏡インテリ大学院生 |
| `marika` | 茉莉花 | TL全般 | 明るくポップなカフェ店員 |
| `momoka` | 桃香 | TL全般 | 2児の主婦 |

#### 感情モード（`novelove_soul.py` の `MOOD_PATTERNS` で定義）
リライト時に指定可能なモード例：
- `"今回は特にこの作品に心をつかまれている。いつもより熱量が高い。"`
- `"今回は冷静に分析しつつも、総評で本音が漏れてしまう。"`
- `"今回は「まだ知られていない名作」を発掘した興奮がある。布教欲が強い。"`
- `"今回はじわじわと効いてくるタイプの感動を受けている。噛みしめるように語る。"`
- `"今回は開始数ページで心を持っていかれた衝撃がある。勢いのまま語る。"`
- `"今回は「こういう作品を待っていた」という運命的な出会いを感じている。"`
- 指定しない場合はランダム選択。

#### ライター・感情モードのオーバーライド（`auto_post.py` 拡張）
- `generate_article()` にオプション引数 `override_reviewer_id` / `override_mood` を追加（A案で確定）
- 引数が指定されていればランダム選択をスキップし、その設定で執筆する
- **既存の自動投稿フローには一切影響しない**（引数なしの場合は従来通りランダム）

#### リライト時のタグ再構築
リライト時は SEOタイトル・メタディスクリプション・AIタグ・完成品タグ（wp_tags）をすべて最新AIで再生成し、WPとDBの両方を更新する。

**セール/売れ筋タグの保護ロジック**：
1. WP REST API で対象記事の**現在のタグID一覧**を取得
2. セールタグ（slug: `sale`）と売れ筋タグ（slug: `best-seller`）のIDを**保護リスト**に退避
3. 通常タグ（サイト名、AIタグ、担当者名）だけを `auto_post.py` と同一ルール（`exclude_list` フィルタ含む）で再構築
4. 保護リストと合体してWPに一括送信
→ `nexus_revive.py` が管理するセール/売れ筋タグと競合しない

#### DB更新（リライト成功時）
- `ai_tags`: 新しいAI生成タグで上書き
- `wp_tags`: 実際にWPに送信した完成品タグ一覧で上書き
- `reviewer`: 今回の担当ライター名で上書き
- `rewrite_count`: +1 加算（リライト回数の累積記録）
- `desc_score`: **変更しない**（取得元のあらすじ充実度であり、記事品質の指標ではないため）

#### CLI対応（ダッシュボードがなくても実行可能）
```
python nexus_rewrite.py --product-id RJ01570022 --reviewer shion --mood "布教欲が強い"
python nexus_rewrite.py --product-id RJ01570022  # ライター/感情はランダム
```

### あらすじ更新検知（敏感モード）

公開済み記事の取得元サイトで、あらすじ情報が追加・変更された場合に検知し、ダッシュボードに通知する機能。

#### 検知ロジック
- `nexus_revive.py`（1日2回巡回）に相乗りし、公開済み記事の公式あらすじを再取得
- DBの既存 `description` と再取得したあらすじを `difflib.SequenceMatcher` で比較
- **内容に変化があれば即座に `is_desc_updated = 1`** をセット（文字数の閾値ではなく、内容変化があれば検知する敏感設計）
- **旧あらすじは `prev_description` カラムに退避**し、新しいあらすじで `description` を上書き（差分ビュー表示のため両方を保持する）

#### ダッシュボード上の表示
- 記事一覧に「📝あらすじ更新あり」バッジを表示
- バッジをクリックすると**差分ビュー**を展開：
  - 追加された文字数
  - 変更率（%）
  - 具体的にどの部分が増えた/変わったかをハイライト表示（`prev_description` と `description` の diff）
- ユーザーが「確認済み」を押すと `is_desc_updated = 0` にリセットされる

v12.8.0にて整備された `wp_tags` カラムを活用し、ダッシュボード上から「実際にWordPressへ送信されているタグ」を直接手動編集・一括置換する機能。

## フェーズ3: Google Search Console（サチコ）連携

### 目的
インデックス状況やアクセスデータに基づいて、**「死に記事」をシステムが自動で検知・通知**し、リライト対象の優先順位を実データで判断できるようにする。

### 連携方法
- Google Search Console API を使用（サービスアカウント認証）
- 実装ファイル: `nexus_gsc.py`（新規作成）
- DB に以下のカラムを追加：
  - `gsc_indexed` (INTEGER): インデックス登録済みか（0/1）
  - `gsc_impressions` (INTEGER): 直近30日間の表示回数
  - `gsc_clicks` (INTEGER): 直近30日間のクリック数
  - `gsc_last_checked` (TIMESTAMP): 最終チェック日時
- 定期巡回（Cron / 1日1回）でデータを自動更新

### 「死に記事」の自動検知＆通知（自動）
サチコのデータを使い、以下の条件に該当する記事を**システムが自動で「死に記事」と判定**する：
- **レベル1（危険）**: 公開後30日以上経過しているのにインデックス未登録
- **レベル2（要注意）**: インデックス済みだが、直近30日間の表示回数が0
- **レベル3（低パフォーマンス）**: 表示はあるがクリック数が0（タイトル/抜粋に魅力がない可能性）

検知された記事は：
- ダッシュボード上に **「⚠️ 死に記事アラート」** として自動表示（レベル別にバッジ色を変える）
- **Discord日次サマリー**に含める（「死に記事: レベル1が○件、レベル2が○件、レベル3が○件」のように件数を通知）

### リライト判断と実行（ユーザー主導）
死に記事の検知・通知まではシステムが自動で行うが、**「この記事をリライトする」という判断と実行はユーザーが行う**。
ダッシュボードのアラート画面から直接リライト確認画面へ遷移でき、ワンクリックでリライトフローに入れる導線を用意する。
