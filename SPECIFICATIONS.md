## 🧭 第1章：システム・アーキテクチャ概要

Noveloveは、主要配信プラットフォーム（FANZA / DMM / DLsite / Lovecal）から新着コンテンツをリアルタイムに収集し、DeepSeek V4-Flash（直接API / OpenRouterフォールバック）による審査と執筆を経て、WordPressへ自動投稿する自律型エンジンです。
- **DLsite一般同人（全年齢）の対応 (v20.0.0)**: DLsiteの全年齢向け漫画・小説・ボイスを自動巡回する機能を追加し、アフィリエイトURLの生成で自動的に `floor="home"` になるよう判定を追加しました。
- **DigiKet関連コード・定義の完全削除 (v20.0.4)**: WordPress上の過去記事完全削除に伴い、これまで既存記事サポートのために維持していたDigiKet関連のコード（アフィリンク生成、スクレイピング関数、関連定数等）および設定定義を完全に削除しました。

- **取得と審査の完全分離**: 取得した作品はすべて在庫（`watching`）として確保し、投稿ループの直前に品質判定と執筆・タグ生成を同時に行うことでAPIコストを抑えます。
- **2択の厳密な判定**: 作品は「合格（投稿）」か「除外（不採用）」のみで判断し、保留ステータスは設けません。情報の薄い作品（スコア3未満）やノイズは自動で除外されます。
- **執筆エラーからの復帰機構**: AIの執筆エラー（一時的な通信障害など不慮の事故）が発生した場合は即座に破棄せず、`pending` のまま次回サイクルで再挑戦させます。
- **在庫の自動鮮度管理**: 取得から7日以上経過した古い在庫や、ジャンルごとの上限（60件）を超えローカル期待値スコアの低いものは自動で破棄されます。
- **投稿インターバル制御**: 全サイト・全ジャンル共通で最短25分の投稿インターバルを設け、1日およそ48件というペースを担保します。

## 🔍 第2章：データ取得と初期選別 (Fetcher)

`novelove_fetcher.py` にて、外部サイトからの情報収集と初期フィルタリングを統括します。

### 2-1. データソースとアプローチ
- **多段取得設計**: APIベース（DMM / FANZA / Lovecal）と、安定したスクレイピングベース（DLsite）を併用します。DMM系はAPIのフロア指定を用いてセーフサーチの影響を回避します。
- **DigiKet新規取得の停止 (v19.5.0)**: DigiKetは4,479クリックに対しCVR 0.0%（売上・報酬ゼロ）、かつDLsiteとの作品重複率が極めて高かったため、新規取得を停止。ランキング記事の水曜日スケジュールも廃止。その後、v20.0.4にて過去記事の完全削除に伴い全コードを完全クリーンアップしました。
- **旧FANZA商業の完全停止とらぶカルの定義管理 (v19.4.0)**:
  自動投稿システムから旧FANZA商業（成人コミック・小説）を完全排除・取得停止し、過去データも本番DBから物理消去しました。らぶカル（同人漫画、同人小説、ボイス）は成人向け属性を識別し、DMM API等の外部連携やクレジット表示などの規約をシンプルに満たすため、定義上 `site: "FANZA"`の設定値を意図的に一部残していますが、データベース上では `site='Lovecal:r18=1'`、`source_db='lovecal'` に統一して論理的に完全に独立管理されています。
- **らぶカル source_db ミスマッチ修正 (v20.0.2)**: `get_source_db("FANZA")` が `'dmm'` を返す仕様により、らぶカルの自動投稿ラウンドロビンで `source_db` が一致せず投稿が停止していたバグを修正。`FETCH_TARGETS` の各エントリに `"source_db": "lovecal"` を明示指定可能とし、`auto_post.py` の投稿処理で `target_info.get("source_db")` を `get_source_db()` より優先適用するよう改修しました。
- **堅牢なリトライ機構**: 502エラーやAPIのレート制限（429）に対し、指数バックオフを用いた自動再試行機能を提供します。
- **あらすじ本文の最大化取得**: APIの文字数切り詰め（特にDMM系の末尾切断）を防ぐため、HTMLクラス解析および公式JSON-LD形式での個別スクレイピングを自動フォールバックで並行処理し、最も文字数の多いものを正規採用します。
- **メタデータ相補的2重スキャン（DLsite）**: DLsiteの遅延描画による取得漏れを防ぐため、`#work_maker`（ブランド/出版社）と `#work_outline`（クリエイター情報）の2つのテーブルを上から網羅的にスキャンしてマージします。また、UIボタンテキスト（「フォローする」「お気に入り」等）の自動除外フィルタを適用し、常にクリーンなデータのみを保存します。
- **DMM一般商業・同人メタデータの網羅**: DMM APIからの商品詳細取得時、`iteminfo` から一般商業の `label`（レーベル）や `maker`（出版社）も抽出し、重複のないようにスクレイピングデータとマージして保存します。らぶカル同人については、`informationList__ttl`（見出し）と `informationList__txt`（内容）から「作者」「シナリオ」「イラスト」等の詳細属性を正確に引き当てます。
- **スペックブロックのグループ化と重複排除・完全パース (v21.2.5)**:
  `build_specs_html` の著者詳細（`author_detail`）のパース処理を最新化。同一クリエイターが複数役割を持つ場合のグループ化表示（中黒 `・` 結合）を引き続き行いつつ、新たに以下の強力なパース仕様を実装してバグを完全に排除：
  1. **役割の引き継ぎ（コロンなし対応）**: `著者:歩くゼリー,イアキ,マンテギ` のように役割名が省略されたカンマ区切りの文字列が含まれる場合、直前の役割名（この場合は `著者`）を自動的に引き継ぐことで、本番表示からクリエイター名が消滅するのを防止。
  2. **役割名の厳格なバリデーション**: `VALID_ROLES` に定義された正当な役割名（`著者`,`サークル`,`出版社`,`レーベル`,`シナリオ`,`イラスト`,`声優(CV)`,`原作`,`WA`）のみを抽出し、無効な役割名（例: `発売日`）やコロンのないゴミパーツを排除。
  3. **システムノイズ排除**: 値に「情報」や「掲載終了」を含むパーツを自動スキップ（配信停止・販売終了時のノイズ表示防止）。
  4. **1人多役の中黒合体**: 同一クリエイターが複数役割を担う場合、`著者・イラスト: 〇〇` のように中黒で結合（会社名である出版社・レーベル・サークルは合体対象外として単独表示）。
  5. **順序の固定化**: 表示順を `著者 ➔ シナリオ ➔ イラスト ➔ 原作 ➔ WA ➔ サークル ➔ 出版社 ➔ レーベル ➔ 声優` に固定化し、統一感のあるスマートなデザインを実現。

- **あらすじ取得とランキングの競合回避 (v19.3.1)**: 通常のあらすじ抽出（`scrape_description`）では漫画・小説以外の形式（ボイスなど）が除外されますが、ランキングコンテキストでの呼び出し時には `is_ranking=True` を明示的に渡すことで、除外処理を安全にスキップさせ全形式を許容します。引数が省略された場合でも従来のジャンル名推論（`genre in ("BL", "TL")`）がフォールバック動作し、後方互換性が担保されます。

### 2-2. データフィルタリング
- **属性による足切り**: 「ゲーム」「動画」「外国語版」「男性向け」は初期段階で確実に取り除かれます。「ボイス/ASMR」は**v19.0.0以降、女性向け（BL/TL）に限り取得対象**に復活しました。漫画・小説ターゲット時はボイス作品を従来通り除外し、ボイスターゲット時のみ取得します。
- **薄い作品の除外**: 単話・分冊・プチ等を示す特定のキーワードを含み、かつ「50ページ未満」の小ボリューム作品は個別記事の質を担保するため自動スキップされます。（例外として週間ランキング記事ではこの制限は適用されません）

### 2-3. 画像取得と表示の最適化 (A+C戦略)
ページの読み込み速度とサーバー負荷軽減のため、アイキャッチ用と本文用でURLを使い分ける高度な軽量化システムです。
- **アイキャッチ（FIFU用）**: DLsiteの `300x300` 版や、DMM of the DMMの軽量版（`ps.jpg`）を採用し数KB単位まで自動軽量化します。※DigiKet画像・処理はv20.0.4で完全削除済み。
- **記事本文用**: オリジナルの高解像度画像をHTMLの `<img>` 要素として大々的に埋め込みます。

### 2-4. 作品基本情報の自動パース (v21.2.0)
通常記事のファーストビューにスリムなスペック表を挿入するため、情報収集時に以下のスペック項目を自動的に抽出して `novelove_posts` テーブルの新規カラムに保存します。
- `author_detail` (TEXT): 著者詳細（例: "著者:〇〇,イラスト:▲▲"）。DLsiteのスペック表、またはDMM APIの `author`, `writer`, `artist` から抽出。
- `cast_info` (TEXT): 出演声優キャスト情報（カンマ区切り）。DLsiteのスペック表、またはDMM APIの `actress` から抽出。声優がいない作品は空文字。
- `series_name` (TEXT): シリーズ名。DLsiteのスペック表、またはDMM APIの `series` から抽出。（※シリーズ名はデータ収集は行われますが、表のスリム化のため自動表示からは除外されています）
- `page_count` (INTEGER): 作品のページ数（ボイス作品の場合は音声本数）。DLsiteのスペック表、またはDMM API of `volume` から数値を抽出。

## 🖋 第3章：AI執筆エンジンと生成ロジック (Writer)

`novelove_writer.py` と `novelove_soul.py` にて、OpenRouter経由のAIによる「生きた言葉」での記事作成を行います。0〜5 段階のスコアを算出し、スコア4以上のみを執筆対象とします。現在、AIモデルには **DeepSeek-V4-Flash**（`deepseek-v4-flash`）を全記事の審査・執筆に統一使用しており、極限のコスト最適化と高い推論クオリティを両立させています。

- **APIフォールバック (v17.7.1)**: DeepSeek直接APIが不通の場合、**OpenRouter経由の `deepseek/deepseek-v4-flash`** に自動フォールバックします。メインと同一モデルを経由するため、フォールバック時も品質低下なし。初回フォールバック発動時にDiscord通知を送信。
- **max_tokens設定 (v17.8.11)**: 執筆時の `max_tokens=16000`。DeepSeek V4 Flashの実際の最大出力は **384,000 トークン**（課金は実際生成分のみ）。この設定は張り切れを防ぐ安全装置として機能し、実際の内容が充分終わればそこで自然に停止する。

### 3-1. ライター（キャラクター設定）— ペルソナ駆動型 (v17.4.0)
以下の5名が、ジャンルごとに異なる視点（「漫画のコマ割り」「小説の心情の機微」など）で作品を解説します。各キャラの `personality` には公式サイト準拠の年齢・職業・背景・好み・苦手を分厚く定義し、AIに自由な語彙選択を委ねています。
さらに、キャラクター性を自然に強調するため、20%の確率で担当キャラ固有の「口癖（メイン/サブ）」が吹き出しコメント内に1回だけ自然に挿入されるロジックを搭載しています。
- **紫苑**: BL担当 / 28歳IT営業職。執着攻め・年下攻め専門。伏線や関係性の構造を冷静に読み解く大人の分析派（口癖: 解釈一致 / 沼が深い…）
- **葵**: BL担当 / 22歳理系大学生。コミケ猛者で作者へのリスペクトが厚い。限界オタク（口癖: 作者さんわかってる…！ / 語彙力が死んだ）
- **蓮**: BL担当 / 24歳文学研究科院生。「あくまで研究」と言い張る隠れ熱血派。学術的に考察（口癖: …個人的な感情ではなく / 構造が美しい…）
- **茉莉花**: TL担当 / 24歳カフェ店員。「全人類ハッピーエンド」が合言葉。全力で共有したがるポジティブ派（口癖: これ、ヤバい… / 心臓もたない）
- **桃香**: TL担当 / 32歳専業主婦。深夜のひとり時間にTLを嗜む。執着を大人の目線で深く語る（口癖: わかりみが深い / この執着がたまらないのよね）
※ 10%の確率で、特定の担当ジャンルを持たない「専門外のゲスト」が選出され、新鮮な角度からの記事を作成します。

### 3-2. HTML骨格パターンの多様化 (Scaled Content Abuse対策)
検索エンジンの「テンプレート量産」判定ペナルティを回避するため、記事形式（DOM構造）はスコアとあらすじ情報量に応じて**「完全に異なる4パターン」からランダム選出**されます。
※一度選出されたパターンは、NGフレーズ等による再執筆（リトライ）が発生しても堅持されます。

**【パターンA（リスト型）】** 従来の基本形式。見どころ・おすすめポイント等の章で箇条書き(`ul/li`)を使用。
**【パターンB（深掘り型）】** `ul/li` 絶対禁止。h3見出しを立てて各魅力を段落単体で長々と解説。
**【パターンC（Q&A型）】** `ul/li` 絶対禁止。読者の疑問に対する「一問一答形式」で展開。
**【パターンD（引用型）】** `ul/li` 絶対禁止。あらすじ原文を `<div class="novelove-quote">` で引用し、それを紐解く構成（あらすじ300字以上限定）。※Cocoonテーマの疑似要素との干渉を回避するため `<blockquote>` は使用しない。

| セクション展開 | パターンA (ul/li有) | パターンB (h3+段落) | パターンC (Q&A) | パターンD (引用) |
|:---|:---|:---|:---|:---|
| **スコア5 (特大)**<br>本文: 400～700字 | 世界観 / キャラ詳細 / 見どころ(ul)3点 / おすすめ(ul)3点 | 世界観 / キャラ詳細 / 魅力ポイント2点(h3) | 世界観 / キャラ詳細 / Q&A 3問(h3) | 世界観 / キャラ詳細 / 原文引用と解説 |
| **スコア4 (標準)**<br>本文: 300～600字 | 世界観 / 見どころ(ul)3点 / おすすめ(ul)3点 | 世界観 / 魅力ポイント2点(h3) |  世界観 / Q&A 3問(h3) | 世界観 / 原文引用と解説 |
| **吹き出し数** | スコア5: 4個 / スコア4: 3個 | 同左 | 同左 | 同左 |
※スコア3以下は足切り（除外）されるため通常投稿には至りません。

### 3-3. 執筆ガードレール (FACT_GUARD)
1. **ハルシネーションの撲滅**: 「書いていないキャラ名の捏造」や「嘘のストーリーの創作」は絶対禁止。あらすじテキスト内から読み取れる情報を唯一の絶対的ソースとさせます。
2. **NGフレーズの禁止**: AIブログ特有の陳腐なクリシェ（「いかがでしたか」「ぜひ確認してみてください」「必見です」）をプロンプトレベルで一括排除します。
3. **情緒と没入感の担保**: テンプレ化した書き出しを禁止し、世界観の核心やキャラクターの生の感情から始めるように設計されています。また、設定上の「口癖」は毎回必ず連呼させるのではなく、性格理解のヒントとして自然に振る舞わせることでワンパターン化を防いでいます。
4. **段落の可読性確保 (v17.8.5)**: 本文の各セクション（h2直下のpタグ）は1段落120〜180字を目安に複数の`<p>`タグに分割すること。400字以上を1つの`<p>`に詰め込むことを禁止しています。

### 3-4. 出力サニタイズ処理 (v17.8.9)
`generate_article()` 内のサニタイズパイプラインで、AIの不正出力を自動修正します。
- **コードブロック除去**: AIが ` ``` ` で本文を囲んだ場合に自動除去。
- **Markdown見出し変換**: AIが `## 見出し` 形式で出力した場合、`<h2>見出し</h2>` へ自動変換（`###` → `<h3>` も同様）。WPはMarkdownを解釈しないため必須の処理。
- **不正URL除去**: `https?://` で始まるURLを記事本文から全除去（アフィリエイトURL等の混入防止）。
- **⚠️ 重要制約**: 上記URL除去サニタイザーが存在するため、**記事コンテンツ内（`content`変数）に絶対URLを含むHTMLを直接書き込んではならない**。アイコン画像等の固定リソースは必ず相対パス（`/wp-content/uploads/...`）を使用すること。
- **アイコンパス強制修正サニタイザー (v17.8.9)**: AIがアイコンパスを誤出力（`/webp-content/` `/wo-content/` `/webdav/` 等）しても、**キャラクター名（葵|紫苑|桃香|蓮|茉莉花|shion|marika|aoi|momoka|ren）を含む `src` 属性を正規表現で検出し、`/wp-content/uploads/icons/キャラ名.png` に強制上書き**する最終防衛線を搭載。

### 3-5. 通常記事のスペック表自動挿入 (v21.2.1)
通常記事を投稿する際、Cocoonテーマに調和し、スマートフォンのファーストビューを圧迫しない、高さ30px程度のインライン横並びスペックバーを生成し、アイキャッチ画像の直後に自動挿入します。
- **動的フィルタリングと完全パース (v21.2.5)**: データが存在する（空でない）項目のみを動的に抽出し、仕切り線（` ｜ `）で連結します。発売日は本スペック表内に統合して表示されます。声優が存在しない作品等では、声優枠そのものが自動的にスキップされます。ボイス作品（ジャンルに `voice` を含む）の場合は、表示単位を `P`（ページ）から `本`（音声本数）へ自動的に切り替えます（例：`9本`）。また、コロンのないカンマ区切りの著者は直前の役割名を自動継承し、1人多役クリエイターの『中黒・合体』処理を適用。さらにノイズデータ（日付・時刻や『掲載終了』などのシステム側ゴミデータ）はパース時に自動排除します。
- **二重挿入防止ガードレール**: 重複適用を防ぐため、スペック表を `<!-- NOVELOVE_SPECS_START -->` と `<!-- NOVELOVE_SPECS_END -->` のコメントタグで囲みます。また、記事内に既存の「発売日：xxxx/xx/xx」の段落記述があれば重複防止のために自動で除去します。投稿および更新の際は、本文内の古いスペック表ブロックを一旦すべて消去してから、最新のスペック表を再挿入します。
- **挿入位置のフォールバック**: アイキャッチ画像の段落が見つからない場合は、最初の見出し（`<h2>`）の直前に挿入します。それも見つからない場合は、最初の吹き出しの閉じタグの直後、最終手段として記事の先頭に挿入します。

## 🗄 第4章：データベース・在庫・全体制御

Novelove は、システム全体のデータを統合した単一のSQLiteデータベース（`novelove_unified.db`）で状態遷移を管理します。
サイトごとの分割（旧仕様の3DB構成）を廃止し、`source_db` カラム（dmm/lovecal/dlsite）によって論理的に境界を設けることで、重複チェックの堅牢化とダッシュボードの高速化を実現しています。
※ v20.0.4でDigiKetの関連コード・定義を完全削除。

### 4-1. ステータス定義
- `watching`: 取得済み・AI未審査のキュー。（ダッシュボードから復活・再審査させる際の戻し先）
- `pending`: AI審査合格・投稿待ちの有効在庫。
- `published`: WordPressへの投稿完了。
- `excluded`: ノイズやボリューム不足・審査落ちによる除外。
- `deleted`: WP上およびDB上からの論理・物理削除。

### 4-2. 全体制御のアルゴリズム
- **ラウンドロビン巡回**: 全ターゲット（サイト×ジャンル×メディア）のインデックスを記憶し、順番に消化することで特定のジャンルやサイト、メディア形式が優遇・冷遇される事態を防ぎます。
- **専売品割込み優先＋二段階フォールバック巡回 (v21.7.4)**: `pending` 在庫内に専売作品 (`is_exclusive=1`) が存在する場合、ターゲット巡回時に専売品のみを最優先で対象とし、専売品同士でサイト・ジャンル交代（ラウンドロビン）＋品質スコア順 (`ORDER BY desc_score DESC`) で投稿します。万が一専売品の全候補がフィルタ等で除外された場合は自動的に通常全作品を対象としたフォールバック巡回を即時実行し、完売後は標準ラウンドロビン巡回へ復帰します。
- **投稿重複ストッパー**: 記事投稿の直前に全DBおよびWordPressの実環境をチェックし、過去24時間以内の同一タイトル文字列による二重投稿を物理的にブロックします。
- **動的（可変）インターバル制御 (v21.2.6)**: DBの `pending` 在庫数をリアルタイムに取得し、投稿間隔を自動的に3段階で切り替えます。在庫30件以上→14分(実効15分・日最大96件/アクティブモード)、在庫10〜29件→25分(実効30分・日最大48件/標準モード)、在庫9件以下→55分(実効60分・日最大24件/セーブモード)。cronの15分起動間隔がSEOスパム判定の絶対防衛ラインとして機能し、エラー時は25分にフォールバックします。
- **緊急停止ロック（サーキットブレーカー）**: パースエラーやAPI異常による「致命的エラー」が連続発生した場合、`emergency_stop.lock` を生成してプロセス全体を強制停止させDiscordへ救難通知を発行します。

*   **API制限回避と更新の二元管理設計**:

### 6-4. テーマ別まとめ記事システム (`novelove_curator.py`) — v21.0.0
露出の低いタグや作品を優先的にまとめ、「まとめ」カテゴリに内部リンクを集約することでサイト全体のSEO戦闘力を最大化する自動バッチです。

**【テーマ選定ロジック】**
- **週番号ベースのジャンルローテーション**: 第1週=BL、第2週=TL、第3週=交互、第4週=クロスタグ（2つのタグのAND条件）。
- **クールダウン制御**: 90日以内に使用したタグは再選定されない。
- **作品選定**: 該当タグの公開済み記事のうち、GSCクリック数(gsc_clicks)の低い順に5作品を自動選定（露出の低い隠れた名作を優先）。

**【記事構成 (v21.3.12)】**
- **導入コラム**: レビュアーキャラクターがテーマについて語る吹き出し形式のイントロ（DeepSeek V4-Flash生成）。
- **各作品ミニレビュー (3階層構成)**: 
  1. `[セリフ]`（短いなりきりトーク、50〜80字）の吹き出し。
  2. `[魅力見出し]`（目次除外のために `<h3>` ではなく、インライン装飾付きの `<div>` で出力）。
  3. `[解説]`（テーマに特化した詳細、です・ます調の丁寧語で客観解説、120〜150字）。
- **誘導リンクボタン**: 各作品解説の末尾に、あらすじ情報紹介との矛盾を防ぐため自然な **「📖 この作品の個別紹介を見る →」** のテキストリンクボタンを設置。
- **比較テーブル**: 選定作品のスペック（サイト・メディア形式・タグ）を一覧表で表示。内部リンクは同タブ、外部リンクは新タブ+nofollow。
- **フッター**: テーマタグへのリンク（WP REST APIでスラッグを事前取得）。

**【安全機構 (v21.1.0)】**
- **緊急停止チェック**: `is_emergency_stop()` で停止中ならスキップ。
- **排他制御**: `curation.lock` + `main.lock`/`ranking.lock` との排他。1時間超の古いロックは自動解除。
- **AI失敗安全弁**: ミニレビューが2件以上失敗したら投稿を中止しDiscord通知。
- **DB INSERT失敗時通知**: WP投稿成功+DB記録失敗の「孤児記事」を検知。

**【SEO・表示仕様 (v21.3.12)】**
- **目次の完全非表示化**: `the_page_toc_novisible = 1` メタデータを投稿時（WP-CLI経由）に自動付与し、まとめ記事の目次ボックス自体を非表示化（ランキング記事の仕様と統一）。
- **目次の重複・カオス防止**: 魅力見出しのタグを `<h3>` から `<div>` に置換し、仮にテーマ設定等で目次が表示された場合でも目次内が作品タイトルのみでスッキリと整理される構造を担保。
- **メタディスクリプション**: まとめ記事専用テンプレート（120文字制限）。通常記事用の`make_excerpt()`は使用しない。
- **SEOタイトル**: 35文字以内に収める（Google日本語表示枠最適化）。サイト名はGoogleが自動付与するため含めない。
- **H2見出し**: テーマキーワードを含む動的見出し（`「{tag}」のおすすめ{genre}作品{n}選`）。

## 📢 第9章：広告設定とアライメント・デザイン統一仕様 (v21.3.15)

WordPress（Cocoonテーマ）のサイドバー広告枠の構造とアライメント・見出しに関するハイブリッド表示仕様です。

### 9-1. 上段広告枠（本・同人・DMM TVの総合配信）
- **設置位置**: サイドバーの位置3（ナビゲーターの真下）
- **ウィジェットID**: `custom_html` ウィジェット (`custom_html-13`)
- **挿入コンテンツ**: `[novelove_random_dmm]` ショートコード
- **判定と挙動 (画面横幅での動的出し分け)**:
  - ページキャッシュ（bcache）によるスマホ表示時のバグを防ぐため、デバイス判定はサーバー側ではなくJavaScriptの画面横幅（`window.innerWidth <= 1023`）で処理します。
  - **PC表示時 (1023px超)**: DMM TVを除外した**3択**（DMMブックスクーポン 40%、らぶカル 30%、DLsiteがるまに 30%）で抽選。下段のDMM TVとの重複を物理的に防止します。
  - **スマホ表示時 (1023px以下)**: DMM TVも含めた**5択**（DMMクーポン、らぶカル、DLsite、DMM TVアニメ、DMM TV舞台）で均等抽選。
- **DLsite生コードの安全化**:
  - がるまにバナーHTMLは、PHP側で `json_encode()` してJSに引き渡すことで、文字エスケープ崩れによる構文エラー（画面真っ白バグ）を100%防ぎます。

### 9-2. 下段広告枠（DMM TV 専用・PC限定表示）
- **設置位置**: サイドバーの位置4（上段広告の真下）
- **ウィジェットID**: `custom_html` ウィジェット (`custom_html-16`)
- **挿入コンテンツ**: `[novelove_random_dmm_tv]` ショートコード
- **機能**: DMM TV（アニメ総合 / 2.5次元舞台）の2択から1枚を動的ロード。
- **非同期時間差ロード (DMM JS干渉対策)**:
  - DMMの公式広告JS同士が同一ページ内で衝突し、表示が空白化・3連化するバグを防ぐため、下段のロード開始は `setTimeout` を使用して「ページ読み込み完了から300ミリ秒（0.3秒）後」にずらして安全にスタートさせます。
  - 0.3秒のディレイにより、読者には上下ほぼ同時に表示されたように見え、遅延による違和感を完全に解消します。
- **スマホでの非表示制御**:
  - `style.css` のメディアクエリ（`@media screen and (max-width: 1023px)`）内に `#custom_html-16 { display: none !important; }` を書き込み、スマホ表示時は枠ごと完全に非表示にします。

### 9-3. スポンサーリンク見出し（ラベル）と中央寄せの統一
- **フォントデザインの完全一致**:
  - DMM広告の見出しは、Cocoon標準の `.ad-label` クラスを持つDIVタグ（`<div class="ad-label">スポンサーリンク</div>`）を使用します。これにより、サイドバー内の他のウィジェットや見出しとフォントサイズ・カラーが自動的に100%同一になります。
- **完全な中央揃え（センタリング）**:
  - 広告バナーと見出しが左右アライメントで綺麗に中央寄せになるように、ラッパーDIVに対するフレックスボックス配置（`display:flex; justify-content:center;`等）で調整して配置します。
- **キャッシュ管理**:
  - 広告配置の修正更新を反映する際は、画像キャッシュを破損する危険がある `wp cache flush` は使用せず、**KUSANAGIのページキャッシュクリア（`kusanagi bcache clear myblog`）および `systemctl restart php-fpm` のみを実行**します。
