# Novelove システム技術マスタードキュメント (v12.1.0)

本ドキュメントは、Novelove（ノベラブ）自動投稿エンジンの全仕様を網羅した「最上位の設計書」です。他の AI や開発者が、ソースコードを読まずともシステムの全貌を完璧に把握し、保守・拡張ができるように記述されています。

---

## 🧭 1. システム・アーキテクチャ概要

Novelove は、FANZA/DLsite/DMM/DigiKet の4大配信サイトから新着コンテンツをリアルタイムで収集し、DeepSeek AI による「審査」と「執筆」を経て、WordPress へ 1 時間に 1 件のペースで自動投稿する自律型エンジンです。

### 1-1. コア・コンセプト
- **Binary Decision Flow**: ステータスを「合格(pending)」か「除外(excluded)」の二択に絞り、複雑な保留(watching)を排除。
- **Quality First**: AI による 5 段階評価で 4 点以上の高品質コンテンツのみを厳選。
- **Dynamic Tagging (v11.4.0)**: AI が記事内容から最適な SEO タグ（TAGS）を動的に生成し、既存の属性情報を補完。
- **Stock First, Review Later (v11.4.0)**: 取得時の AI 審査を廃止し、まずは全件を `pending` 在庫として確保。投稿タイミングで初めて AI が審査・執筆を行うことで、API コストを最適化。
- **Realism & Engagement (v11.3.5)**: AI 特有の「定型文」や「嘘の創作」を徹底排除し、キャラとしての生の感情を優先。
- **Automatic Inventory Cleanup (v11.4.0)**: 7日以上経過した古い在庫や、ジャンルごとの上限（20件）を超えた在庫を自動で除外。
- **Cross-Platform Sync**: 3 つの SQLite DB を横断管理し、サイト全体で投稿間隔を 55 分以上に制御。

## 🛠 2. 技術コンポーネント詳細

### 2-1. データ取得 (Fetcher & Scraper: novelove_fetcher.py)
- **モジュール分離 (v11.4.3)**: 取得に関連するロジック（API呼び出し、スクレイピング、初期フィルタリング）を `novelove_fetcher.py` へ分離し、本体の軽量化とメンテナンス性の向上を実現。
- **多段取得**: API (DMM/FANZA) および Web スクレイピング (DLsite/DigiKet) を併用。DMM APIではキーワード検索ではなく、直接「カテゴリID（66036等）」および「小説フロア（novel）」を指定することで、セーフサーチの影響を受けず数万件規模の商品群を網羅的に捕捉。
- **コンテンツ・フィルタ**: 取得時に「ボイス」「ゲーム」「動画」「外国語版」「男性向け」を、ジャンルコードやキーワード、HTML バッジ 等で厳密に排除。
- **薄い作品（単話・分冊版）の自動除外 (v11.3.0)**: 個別紹介記事の質を担保するため、以下の「条件を両方満たす」作品を在庫補充時に自動スキップ。
    - キーワード判定: `分冊版`, `単話`, `単話版`, `【マイクロ】`, `【プチ】` のいずれかを含む。
    - ページ数判定: 50ページ未満（API/スクレイピングのメタデータを参照）。
    - ※ランキング記事には本制限を適用せず、全てのボリュームの作品を掲載対象とする。
- **高解像度化**: DigiKet の画像 URL を `_2.jpg` (低) から `_1.jpg` (高) へ自動置換。

### 2-2. AI 執筆プロンプト・エンジニアリング (v11.3.5 刷新)
AI 出力のマンネリ化防止と「AI臭」の排除のため、高度なガードレールと動的感情注入を実装しています。

- **感情モード (MOOD_PATTERNS)**:
    「熱量が高い」「冷静に分析」「布教欲が強い」など 6 種類の感情状態を投稿ごとにランダム選択。冒頭・末尾の定型化を防ぎ、読み手に「意志を持った人間が書いている」印象を与えます。
- **事実性ルール (FACT_GUARD)**:
    AI のハルシネーション（嘘の創作）を物理的に封じ込めるための絶対遵守ルール。
    - あらすじにない「キャラ名」「展開」の捏造を禁止。
    - 実際には読んでいないことを前提とし、「読み終えて」等の偽装表現を禁止。
- **NG表現集 (NG_PHRASES)**:
    「いかがでしたか」「ぜひチェックしてみてください」等の AI ブログにありがちな無難フレーズをプロンプトレベルで一括して使用禁止。
- **多様性の確保**:
    - **書き出しの多様化**: 「本作は～」等のテンプレ開始を禁止し、世界観描写やシーンの切り取りから始めるよう指示。
    - **h2見出しの刷新**: 毎回異なる切り口を要求し、「○○に迫る」等の固定表現を排除。
    - **見どころの並び替え**: 常に「ストーリー・作画・キャラ」の順にせず、作品の強みを優先順位付け。

### 2-3. AI 審査 & 執筆 (Reviewer & Writer Logic - v11.4.0 刷新)
v11.4.0 より、取得時の AI 審査を廃止し、**「投稿直前の 1 回の API 呼び出し」**で審査・執筆・タグ生成を同時に行う方式へ移行しました。
- **Immediate Quality Gate**: 投稿時に AI があらすじを読み、0〜5 点で採点。
- **判定基準**:
    - **4-5点**: 合格。そのまま執筆を継続し、WordPress へ投稿。
    - **1-3点**: 不採用。DB ステータスを `excluded` へ変更（投稿せずスキップ）。
    - **0点 / エラー**: 外国語版や形式不一致。`excluded` へ変更。API 障害時は次回再試行。
- **動的タグ抽出 (TAGS)**: 
    指示に基づき、AI が記事内容から作品に最適なタグを最大 3 つ抽出。DB に保存されている属性タグよりも、この「AI 生成タグ」を優先して WordPress へ反映します。

### 2-4. AI 執筆 (Writer Logic)
- **キャラクター・ペルソナ**: 5 名の個性的なライター（紫苑, 葵, 蓮, 茉莉花, 桃香）を設定。
- **SEO 強化タイトル自動生成 (v12.1.0)**: プロンプトによってAIが記事本文に最適なSEOタイトルを考案します。（ハルシネーションを固く禁じる指示付き）
    - 形式: `『[作品タイトル]』[AIが生成した感情を揺さぶるキャッチコピー]`
    - 文字制限: 68文字（超過時は省略して「…」を付与）
- **紹介文形式メタディスクリプション (v12.1.0)**: あらすじの単なる抜粋ではなく、作品の魅力を担当ライターの視点で紹介。
    - 形式: AIが記事本文を元に、誰のどんな性癖に刺さるかを約80文字の惹き句として出力。
    - 適用: WP-CLIにて `the_page_meta_description` へ直接上書き保存。
- **3段フィルター**: 官能的な記述による API ブロックを避けるため、軽め〜ガチガチの 3 段階マスク処理を試行。
- **HTML 構造**: `<h2>`, `<p>`, `<ul>`, `<li>` を駆使し、吹き出しデザインと標準語の解説文を融合。
    - 本文（`<p>`タグ）の文字数は 200〜400 字程度とし、情報が少ない作品での「こじつけによる文字稼ぎ」を防止。 (v11.3.5)

- **SEO 情報の確実な反映 (v11.1.2)**: 投稿後、SEO タイトルおよびメタディスクリプションを WP-CLI（`wp post meta update`）でデータベースへ直接書き込み。REST API 経由の更新失敗を完全に排除。

### 4-1. ゲスト紹介機能 (10%の確率)
作品に対して担当ライターを割り当てる際、**90%は専門ジャンルのライター**が選ばれますが、**10%の確率で専門外のライターが「ゲスト」として選出**されます。
ゲストとして選ばれた場合、AIプロンプトには「自分の専門外だからこそ気付く新鮮な視点を活かして紹介する」という専用の指示が注入され、サイトに変化と遊び心を与えます。

### 4-2. 2名掛け合い（コラボ）機能 (v12.0.0 追加)
毎週末のランキングまとめ記事では、**必ずメインMCとゲストの2名**が選出され、**交互の吹き出し会話形式**で進行します。5名のライターによる全10通りの関係性マトリクス（例：「限界オタクと見守る母」等）がAIプロンプトに注入され、キャラクター同士の関係性を活かしたリッチでエンタメ性の高い記事が生成されます。

**【ランキング記事の構成仕様（プロンプト指示）】**
ランキング記事の出力は「地の文を一切禁止」し、すべて専用の左右吹き出し（`speech-bubble-left` / `speech-bubble-right`）の対話のみで構成されます。
- **冒頭**: 2人のオープニングトーク。お互いに挨拶し、今週のランキングへの期待を語る（合計4〜6往復）。
- **第5位〜第2位**: 各作品ごとに、あらすじ説明（MC主導）→ゲストのリアクション→推しポイントの掘り下げ（最低3往復）。
- **第1位**: 2名で熱量MAXに語り倒す（最低5往復）。
- **締め**: 今週の感想と読者へのメッセージ（合計3〜4往復）。

## 🗄 5. データベース & 運用

### 5-1. DB ステータス定義
- `pending`: 審査合格（スコア4-5）。
- `published`: 投稿完了。
- `excluded`: 除外（スコア0-3 またはノイズ）。
- `last_error`: `no_image`, `wp_post_failed`, `api_error` 等の理由を記録。

### 5-2. クールダウン & ロック
- **Lock**: `main.lock` / `ranking.lock` による多重起動防止。
- **緊急停止ロック**: `emergency_stop.lock` が存在する場合、全ての処理を最頂部でスキップします。
- **Wait**: 全 3 DB の最新投稿時刻から 55 分経過していない場合、実行をスキップ（時刻計算は JST 準拠）。

### 5-3. 投稿順序と選別ロジックの公平性
- **ラウンドロビン方式**: 全14種の取得ターゲット（サイト×ジャンル）を順番に巡回します。前回投稿したターゲットのインデックス番号を `genre_index.txt` に記憶し、次回はその次のターゲットから再開するため、特定のサイトが優先されることはありません。
- **在庫クリーンアップ (v11.4.0)**: 投稿処理の開始時に、全 DB から以下の条件に合致する `pending` 作品を自動的に `excluded` (expired/inventory_full) へ移動し、在庫の鮮度を維持します。
    - **有効期限**: 取得から 7 日以上経過したもの。
    - **在庫上限**: ジャンルごとに `pending` 在庫が 20 件を超えた場合、古いものからカット。
- **投稿順序 (v11.4.0)**: 在庫（pending）の中から、最も新しく取得された作品（`inserted_at DESC`）を優先的に投稿します。これにより、話題の新着作品が速やかにサイトへ反映されます。
- **投稿の偏りのカラクリ**: 「DLsiteばかりが投稿されている」ように見える現象は、システムがDLsiteをひいきしているのではなく、「他のサイトのpending在庫が尽きている状態で、全サイトを平等に巡回した結果、DLsiteの在庫だけがヒットし続けている」ことが原因です。

### 5-4. サーキットブレーカーと関所システム (Emergency Stop & Cooldown) - v12.1.0 強化
サーバーリソースの枯渇やDBロックエラー、そして無限ループを物理的に遮断するための安全装置群です。

- **自動検知ロジック（緊急停止）**:
    - **10分タイムアウト**: 1回の `_run_main_logic` 処理が 10 分（600秒）を超えた場合、緊急停止を発動。
    - **3回連続エラー**: 投稿試行が 3 回連続で失敗（理由を問わず）した場合、緊急停止を発動。
- **緊急冷却システム（関所）**:
    - `emergency_stop_cooldown(seconds=1)` 関数により、スクリプトの起動時や主要な処理の合間に強制的な待機時間を挟み、同時実行に伴う CPU スパイクや SQLite の `database is locked` エラーを抑制します。
- **発動時の挙動**:
    - サーバー上の実行ディレクトリに `emergency_stop.lock` ファイルを生成。
    - Discord の `🚨 緊急停止通知` チャンネルへ、原因と解除方法を即時通知。
    - 以降、cron 等でスクリプトが起動しても、ロックファイルが削除されるまで一切の処理を行わずに終了。
- **解除方法**:
    - サーバーに SSH 接続し、手動で `rm /home/kusanagi/scripts/emergency_stop.lock` を実行する。

## 🏷 6. 抽出・分類詳細定義 (v10.6.0 刷新)

ノベラブ独自の分類ルールに基づき、記事を「カテゴリ（形態）」と「タグ（属性）」に分割して整理します。

### 6-1. カテゴリ定義（4大分類）
作品の「ジャンル」と「メディア形態」に基づき、以下の4つのカテゴリのいずれかに分類します。
- `BL漫画` (bl-manga) / `TL漫画` (tl-manga) / `BL小説` (bl-novel) / `TL小説` (tl-novel)

> [!IMPORTANT]
> **スラグ（URL）の完全英名化ルールの徹底**
> WordPressにおけるカテゴリ・タグ等のスラグ（URLの一部）は、**いかなる場合でも日本語を使用せず、すべて英単語またはローマ字（例: `omegaverse`、`doujin`）で統一**します。これは SEO 評価の最大化および URL 共有時の文字化けを防ぐための絶対的な仕様です。

### 6-2. サイト別・抽出判定ロジック
| サイト | 商業/同人の判定 | 小説(Novel)の判定 |
| :--- | :--- | :--- |
| **DMM/FANZA** | `service` が `ebook` なら **商業** | API の `floor_code` が `comic` → 漫画 / `novel` → 小説。同人（`digital_doujin`）は `iteminfo.genre` のジャンル ID 115 が含まれれば小説、それ以外は同人漫画と確定。タイトルキーワードは**一切参照しない**。|
| **DLsite** | URLに `/bl-pro/` 等があれば **商業** | 詳細ページの公式バッジ `work_type` が `NRE` なら **小説**、`MNG` なら **漫画** と確定。ターゲット設定より公式属性を優先し、タイトルキーワードは**参照しない**。 |
| **DigiKet** | `genre` が `comic_` 系なら **商業** | カテゴリ名に「小説」を含む場合に判定 |

### 6-3. タグ付与のシンプル化ルール (v10.6.0)
データセットのシンプル化と SEO 強化のため、**カテゴリで表現できる情報はタグとして付与しません**。タグは以下の「属性情報」のみに限定します。

- **出処タグ**: `商業` / `同人`
- **年齢タグ**: `R-18` / `全年齢`
- **サイトタグ**: `DLsite` / `FANZA` / `DMM` / `DigiKet`

### 6-4. 週間ランキング記事の特別ルール (v11.2.0 拡張)
ランキング記事（`ranking`）は、複数の異なる属性・ジャンルの作品が混在する「まとめ記事」であり、以下の特別ルールで運用されます。
- **統合カテゴリ**: 「BL総合（漫画＋小説）」「TL総合（漫画＋小説）」の2大ジャンルに集約。
- **配信スケジュール**: 
    - 日曜: FANZA / 月曜: DLsite / 火曜: DMM.com / 水曜: DigiKet
    - 毎日 **22:00 / 22:30** の2回投稿（15-30分間隔）。
- **付与するタグ**: 「配信サイト名（DLsite等）」と「担当ライター名（紫苑等）」**のみ**付与。
- **付与しないタグ**: 出処（商業/同人）、年齢制限、AI属性タグは一切付与しない。

### 6-5. 特定ターゲットの除外理由（DMM同人等）
システムは「最も良質な作品が豊富にある場所」を効率的に巡回するため、以下のようにターゲットを意図的に絞っています。
- **DMM.comの同人作品を除外している理由**: R-18の同人作品はすべて「FANZA」へ集約されており、全年齢（DMM.com）の同人作品は極めてニッチで少数です。そのため、全年齢同人は最大手の「DLsite」と「DigiKet」に任せ、DMM.comは「一般商業（プロ作品）」の取得に特化させています。これにより、AI審査で低得点になるノイズ作品の取得を減らし、合格率を高めています。

### 6-6. 小説（ノベル）機能の実装状況 (v11.0.0 完了)
DLsite（同人・商業）、DMM（商業）、FANZA（商業・同人）、DigiKet（同人・商業）の全4プラットフォームにおける「小説（ノベル）」の自動取得・審査・投稿に完全対応しています。
過去のストックは対象とせず、各サイトの「新着順（新刊）」に特化してリストを取得し、漫画と同様にラウンドロビン方式の巡回対象として `FETCH_TARGETS` に組み込まれています。

### 6-7. 統合ランキング機能 (v11.2.0 追加)
4プラットフォームすべての週間ランキングから、漫画と小説を統合した「BL総合」「TL総合」ランキングを生成します。
- **独立記事生成**: 同一サイトであっても、ジャンル（BL/TL）ごとに完全に独立した **別個の記事** として生成・投稿されます。
- **メディア統合**: 漫画と小説を売上・人気順に基づき抽出し、1つのランキング（TOP5）としてバランスよくマージします。
- **内部リンク構成**:
    - **紹介記事連携**: ランキング内の各作品に対し、DB内に過去の個別紹介記事が存在する場合は「📝 詳しい紹介記事はこちら」というリンクを自動挿入します。
    - **相互リンク**: BL記事の末尾には「TL総合ランキング」へのリンクを、TL記事の末尾には「BL総合ランキング」へのリンクを自動挿入し、回遊性を最大化します。

### 6-8. 週間ランキング投稿スケジュール
投稿が集中して埋没するのを防ぐため、以下の曜日に分散して実行されます（毎日 22:00 開始）。
- **日曜日**: FANZA 総合ランキング（BL/TL）
- **月曜日**: DLsite 総合ランキング（BL/TL）
- **火曜日**: DMM.com 総合ランキング（BL/TL）
- **水曜日**: DigiKet 総合ランキング（BL/TL）

### 6-9. 重複防止ガードレール (v11.4.13 強化)
「サイトの信頼性」を守るため、同一作品が二重に投稿されることを物理的・論理的に防止します。

- **投稿直前のタイトル監査**:
    投稿の数秒前に、全 3 つの DB（FANZA/DLsite/DigiKet）の `published` レコードおよび WordPress 上の全公開記事から「タイトル文字列」を検索。
    - 同一タイトルが過去 24 時間以内、または同一 ID が存在する場合、ステータスを `excluded` (duplicate) に変更して投稿を中止。
- **例外発生時の強制除外とクエリ最適化**:
    記事生成や画像設定中に予期せぬエラーが発生した場合、`try-except` ブロックで確実に捕捉し、該当作品を `pending` から `excluded` へ移動。また、SQLite の特有エラー（`1st ORDER BY term out of range`）を防ぐため、動的SQLの構築ロジックは定数を使わず厳格なパラメータバインド順序で安全にソートされます。

## 🛠 9. 過去記事のメンテナンスとデプロイ手順

### 9-1. スクリプトの転送
Windows環境からサーバーへは **`scp` コマンド**を推奨。

### 9-2. 過去記事タグ一括再編スクリプト (`cleanup_v10_6.py`)
既存の全記事から不要なタグ（BLコミック等）を削除し、新しい大分類カテゴリ（BL漫画等）と属性タグ（商業等）に置き換えるためのクリーンアップを実行。

### 9-3. WP-CLI を活用した高速メンテナンス
大量の記事データを一括修正・更新する場合、REST API 経由ではなくサーバー上での **WP-CLI** 実行を推奨します。

- **利点**: HTTP 通信のオーバーヘッドがなく、数千件のデータ処理を数秒〜数十秒で完了可能。
- **推奨ワークフロー**:
  1. **検証**: まずは 1〜5 件の記事に対して API や手動で修正を行い、表示に問題がないか慎重に確認する。
  2. **一括実行**: ロジック確定後、WP-CLI コマンド（例: `wp post meta update`）や `wp eval-file` を用いてサーバー上で一括処理を行う。

## 📁 10. ファイル構成と役割分担

### 10-1. 基本構成
| ファイル名 | 役割 | 編集が必要なケース |
| :--- | :--- | :--- |
| `auto_post.py` | ▶ **本体エンジン（司令塔）** | 戦略（投稿フロー制御・執筆プロンプト・ランキング進行）の変更 |
| `novelove_fetcher.py`| 🔍 **取得（外部接続）** | 新着取得・スクレイピング・フィルタリング・在庫補充ルールの変更 |
| `novelove_soul.py` | 🎭 **魂（人格）** | ライターの性格・口癖・NGワードの変更 |
| `novelove_core.py` | 🛠 **インフラ（内臓）** | DB接続・通知・基盤処理（通常時編集不要） |
| `SPECIFICATIONS.md` | 📚 **設計書** | 本仕様書の更新 |
| `CHANGELOG.md` | 📝 **変更履歴** | バージョンアップ時の記録 |
| `novelove.db` 等 | 🗄 **データベース** | 自動更新（手動編集不要） |

### 10-2. novelove_fetcher.py の役割詳細
`novelove_fetcher.py` は、外部サイト（FANZA, DLsite, DMM, DigiKet）からのデータ取得と、一次フィルタリングを担当するモジュールです。

- **含まれる内容**: `FETCH_TARGETS`（取得先定義）、各種スクレイピング関数、R-18/薄いコンテンツ等の判定ロジック、スクレイピング用セッション管理。
- **ポイント**: **「どうやってデータを集めるか」** を管理します。サイトの仕様変更があった場合は、主にここを修正します。

### 10-3. novelove_soul.py の役割詳細
`novelove_soul.py` は、Novelove の AI ライターたちの「人格」に関する設定を集約した専用モジュールです。

- **含まれる内容**: `REVIEWERS`（5名のライター設定）・`MOOD_PATTERNS`（感情モード）・`FACT_GUARD`（事実性ルール）・`NG_PHRASES`（禁止フレーズ）
- **ポイント**: AI の「しゃべり方」だけを管理します。

### 10-4. novelove_core.py の役割詳細
`novelove_core.py` は、プログラムが動くための「機械的な仕組み」を管理するインフラモジュールです。

- **含まれる内容**: データベース接続、Discord通知ロジック、画像サイズチェック、文字列整形ユーティリティ。
- **ポイント**: **「どうやって動かすか」** を管理します。一度安定すれば、運用者がここを編集することはまずありません。
- **メリット**: 本体の `auto_post.py` から難解なコードを分離することで、メインエンジンの可読性を劇的に向上させています。
