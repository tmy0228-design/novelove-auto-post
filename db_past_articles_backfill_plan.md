# 既存記事への属性・専売タグ反映（バックフィル）計画書

今回の `v12.7.0` アップデートにより、新規に取得する作品データには「公式属性タグ（`original_tags`）」および「専売フラグ（`is_exclusive`）」が正しくデータベース（`novelove_posts` テーブル）に保存され、WordPress投稿時にこれらが適用されるようになりました。

しかし、**既に投稿済みの過去記事（約〇千件）や、現在 `pending` 等の状態でDB内にストックされている既存データには、これらの新しいカラム情報が空（またはデフォルト値）のまま**となっています。

過去記事に対しても同等のSEO効果とアフィリエイト購入転換率（コンバージョン率）向上の恩恵をもたらすため、以下のプロセスでデータベースの遡及更新とWordPressへの一括反映（バックフィル）を実施する予定です。

---

## 1. 目的と期待効果

1. **転換率の底上げ**: 過去の優良記事にも「DLsite専売」「FANZA独占」「DigiKet限定」などの限定販売訴求タグを付与することで、ユーザーの「ここでしか買えない」という心理を喚起し、購入へのコンバージョン率を高める。
2. **SEO・サイト内回遊の改善**: 公式の「属性タグ（オメガバース、純愛など）」から適正に抽出されたタグを過去全記事に統一して付与し直すことで、タグページの充実とサイト内のハブ・スポーク構造を強固にする。

---

## 2. 作業フェーズ案

全件を手作業で再生成すると膨大なAPIコストと時間がかかるため、システム的かつ段階的に反映させます。

### フェーズ 1: データベース（SQLite）の空カラム再取得
既に取得済みの `product_id` のリストをもとに、各プラットフォームから「属性タグ」「専売フラグ」のメタデータのみを再収集し、DB上の `original_tags` と `is_exclusive` を埋める専用のパッチスクリプトを作成・実行します。

* **対象**: `novelove_posts` において `is_exclusive` が 0 かつ `original_tags` が空文字のレコード群。
* **動作**: 既存の `novelove_fetcher.py` のロジック（`scrape_dlsite_description`等）を再利用。URLにアクセスしタグ部分のみを取得し、UPDATE文で書き換えます。`description` や画像などの再取得はスキップして高速化します。

### フェーズ 2: 既存のWPタグ一括アップデート（専売タグ付与）
メタデータ補完が完了したDB情報に基づき、WordPressに公開済み（`status='published'`）の記事に対して、「〇〇専売」「〇〇独占」等のWPタグのみを付与します。

* **対象**: DB上で `is_exclusive = 1` と判明した過去記事。
* **動作**: WP-CLI, またはWordPress REST API を用い、該当記事の既存タグ配列に対して「DLsite専売」等をマージしてアップデートします。
* **安全性**: 本文やタイトルは一切変更しない（非破壊）ため、検索順位を落としたりHTML構造を壊すリスクはありません。現在の `nexus_revive.py` のセールタグ付与と同じクリーンな手法を採用します。

### フェーズ 3: （オプション）過去記事のリライトとAIタグ再生成
一部のアクセス上位記事や、以前のAIによって内容が薄く書かれてしまった「スコア3病」の記事を対象に、今回新設した「属性タグを読み込ませるプロンプト」を使って記事自体をリライト（再執筆）させます。

* **対象**: `desc_score <= 3` の公開済み記事、またはGoogle Search Console等で表示回数が多いがクリック率が低い記事。
* **動作**: 新プロンプトで生成した高品質なコンテンツで WordPressの本文を上書き更新します。タグリストも最新のAI生成のものを適用します。
* **コスト**: OpenAI / DeepSeek のAPI利用料が再発生するため、一斉に行うのではなく1日100記事などのペース配分を設けます。

---

## 3. 実装の注意事項とNext Action

* **既存コードとの重複を避ける**: スクリプト作成時は `novelove_fetcher.py` 内部に専用のバルクターミナルコマンド（例: `python novelove_fetcher.py --backfill`）を作るか、独立した `backfill_tags.py` を書くかを検討します。
* **APIレート制限対策**: 一度に数千件のURLをスクレイピングするとDDoSと判定されるため、スクレイパーには 1.5秒 〜 2.0秒 の厳格な `time.sleep()` を設定し、一晩〜数日かけてゆっくりデータを埋める設計とします。
* **実行環境**: 本番サーバー上（XSERVER VPS）で `tmux` セッションを使用してバックグラウンドでロングランさせます。
